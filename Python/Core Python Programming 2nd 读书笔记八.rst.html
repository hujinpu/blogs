在Python 2.5之前一直没有了三元运算符，如果要返回两个数中较大的，一种Hack的方法是：

>>> x, y = 3, 4
>>> x
3
>>> y
4
>>> smaller = (x < y and [x] or [y])[0]
>>> smaller
3

这里要用 [x] 和 [y] 集合的目的是为了避免x和y本身为等同False的值

那么到了Python 2.5， Guido van Rossum 最终选择了一个最被看好(也是他最喜欢)的方案，加入了三元运算符，语法确定为: X if C else Y ，做同样事，现在更加重要简洁了。

>>> smaller = x if x < y else y
>>> smaller
3

迭代序列有三种基本方法：

  • 通过序列项迭代：

    >>> nameList = ['Walter', "Nicole", 'Steven', 'Henry']
>>> for eachName in nameList:
...     print eachName, 'Lim'
...
Walter Lim
Nicole Lim
Steven Lim
Henry Lim

  • 通过序列索引迭代，没有上一种性能好：

    >>> nameList = ['Walter', "Nicole", 'Steven', 'Henry']
>>> for nameIndex in range(len(nameList)):
...     print 'Liu,', nameList[nameIndex]
...
Liu, Walter
Liu, Nicole
Liu, Steven
Liu, Henry

  • 同时使用序列项和索引迭代：

    >>> nameList = ['Donn', 'Shirley', 'Ben', 'Janice', 'David', 'Yen', 'Wendy']
>>> for i, eachLee in enumerate(nameList):
...     print '%d %s Lee' % (i+1, eachLee)
...
1 Donn Lee
2 Shirley Lee
3 Ben Lee
4 Janice Lee
5 David Lee
6 Yen Lee
7 Wendy Lee

xrange() 类似 range() ，不过当你有一个很大的范围列表时， xrange() 性能更好，因为它不会在内存里创建列表的完整拷贝。在Python的将来版本中， range() 可能会像 xrange() 一样，返回一个可迭代对象(不是列表也不是一个迭代器)。

注意： sorted() 和 zip() 内建函数返回的是列表，而 reversed() 和 enumerate() 内建函数返回的是迭代器。

在Python中，对于while和for循环，可以在循环后加入else语句，只要是正常退出的循环都会执行else语句里的代码块，如果break退出的循环就不会执行了。

为什么需要迭代器呢？

  • 提供了可扩展的迭代器接口
  • 对列表迭代带来了性能上的增强
  • 在字典迭代中性能提升
  • 创建真正的迭代接口, 而不是原来的随机对象访问
  • 与所有已经存在的用户定义的类以及扩展的模拟序列和映射的对象向后兼容
  • 迭代非序列集合（例如映射和文件）时，可以创建更简洁可读的代码

根本上说，迭代器就是有一个 next() 方法的对象，当你或是一个循环机制（例如for语句）需要下一个项时，调用迭代器的 next() 方法就可以获得它。条目全部取出后，会引发一个 StopIteration 异常，这并不表示错误发生，只是告诉外部调用者迭代完成。

不过，迭代器也有一些限制。例如你不能向后移动，不能回到开始，也不能复制一个迭代器。如果你要再次（或者是同时）迭代同个对象，你只能去创建另一个迭代器对象。

一个for循环实际是这样工作的：

fetch = iter(seq)
while True:
    try:
        i = fetch.next()
        do_something_to(i)
    except StopIteration:
        break

一个序列的迭代器只是记录你当前到达第多少个元素，所以如果你在迭代时改变了元素，更新会立即反映到你所迭代的条目上，注意迭代器是和序列对象绑定在一起的。

>>> a = range(10)
>>> for i in a:
...     print i
...     a.remove(i)
...
0
2
4
6
8
>>> a
[1, 3, 5, 7, 9]

在迭代字典的key时，你绝对不能改变这个字典。使用字典的 keys() 方法是可以的，因为 keys() 返回一个独立于字典的列表。

>>> myDict = dict(a=1, b=2, c=3, d=4)
>>> for k in myDict.keys():
...     print k, myDict[k]
...     del myDict[k]
...
a 1
c 3
b 2
d 4
>>> myDict
{}

但直接使用字典就不行了，迭代器是与实际对象绑定在一起的，它将不会继续执行下去。

>>> myDict = dict(a=1, b=2, c=3, d=4)
>>> for k in myDict:
...     print k, myDict[k]
...     del myDict[k]
...
a 1
Traceback (most recent call last):
  File "<stdin>", line 1, in <module>
RuntimeError: dictionary changed size during iteration

Python这么做是为了避免有缺陷的代码。

列表解析的表达式可以取代内建的 map() 函数以及 lambda ，而且效率更高。

这是扩展的版本： [expr for iter_var in iterable if cond_expr]

一个有三行五列的矩阵

>>> [(x + 1, y + 1) for x in range(3) for y in range(5)]
[(1, 1), (1, 2), (1, 3), (1, 4), (1, 5), (2, 1),
 (2, 2), (2, 3), (2, 4), (2, 5), (3, 1), (3, 2), (3, 3), (3, 4), (3, 5)]

再看一个很cool统计文件中非空格字符数的

f = open('data.txt', 'r')
sum([len(word) for line in f for word in line.split()])

更加资料可以在 PEP 202 找到

Python 2.4又引入了一个很好的东西，目的就是为了节约列表解析的内存占用，这就是生成器表达式。它与列表解析非常相似，而且它们的基本语法基本相同；不过它并不真正创建数字列表，而是返回一个生成器，这个生成器在每次计算出一个条目后，把这 个条目“产生”（yield）出来。生成器表达式使用了“延迟计算”（lazy evaluation），所以它在使用内存上更有效。

语法为： (expr for iter_var in iterable if cond_expr)

上面那个例子如果文件很大，那么只是为了求和而构造那么大的数组显然不合理，所以使用生成器就可以避免这个性能的问题：

f = open('data.txt', 'r')
sum(len(word) for line in f for word in line.split())

要找出一个文件中最长的一行，用生成器一切就是这样完美，而且也很易读：

max(len(line)  for line in open('data.txt') if line.strip())

关于生成器的更加说明可以读下 PEP 289
